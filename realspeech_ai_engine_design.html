Section 02-1 · 핵심 아키텍처 이슈

⚠️ STT 보정 Transcript — 무엇을, 어디서 평가하는가

GPT(Ⅱ.내용)에 전달되는 Transcript는 Azure STT의 언어 모델 보정을 거친 텍스트다. 이 사실이 "문법 오류를 어디서 측정하는가"를 완전히 바꾼다.

Azure STT는 2단계로 작동한다

음성 입력 │ ▼ [1단계] 음향 모델 (Acoustic Model) 음소 단위로 소리를 감지 → AccuracyScore / ErrorType 산출 ※ 보정 이전 원시 데이터 — wordDetails 배열 │ ▼ [2단계] 언어 모델 (Language Model) 문맥 기반으로 "가장 그럴듯한 단어"로 보정 ※ 이 결과가 GPT에 전달되는 Transcript 텍스트

오류가 사라지거나 남는 두 가지 케이스

학생이 실제 말한 것음향 모델 (AccuracyScore)GPT가 받는 Transcript결과
"I eated the apple" "eated" 발음 오류 감지 ✅ "I ate the apple"
(언어모델이 "ate"로 보정)		 Ⅰ.언어에서만 반영, GPT는 오류를 못 봄
"I don't know nothing about it" 그대로 인식 "I don't know nothing about it"
(이중부정은 보정 안 됨)		 Azure·GPT 모두 감지 가능 → 이중 반영 위험
보정 여부가 일관적이지 않다. 어떤 오류는 사라지고 어떤 오류는 남는다 — 예측 불가능한 패턴. 따라서 GPT에게 "Transcript에서 문법 오류를 세라"고 시키면 신뢰도가 보장되지 않는다.

scoreLang() — 핵심 로직
function scoreLang(azure) {
  // 1. 의미 전달 방해 오류 — wordDetails(보정 이전 원시 데이터)에서 직접 추출
  //    ⚠️ Transcript 텍스트는 절대 참조하지 않음 (언어모델 보정으로 오류가 사라질 수 있음)
  const meaningBlockErrors = azure.wordDetails.filter(w =>
    w.errorType === 'Omission'   ||  // 단어 누락
    w.errorType === 'Insertion'  ||  // 없는 단어 삽입
    (w.errorType === 'Mispronunciation' && w.accuracyScore < 45)
  );
  const errorRate = meaningBlockErrors.length / Math.max(1, azure.totalWords);

  // 2. AccuracyScore는 음향 모델 단계 측정값 (보정 영향 없음) → 그대로 사용
  const accuracy = azure.scores.accuracy;

  // 3. WPM 이상 시 FluencyScore 상한 캡
  const wpm = azure.wpm;
  const fluencyCap = wpm < 50  ? 55   // 극단적으로 느림
                    : wpm < 80  ? 70   // 매우 느림
                    : wpm > 220 ? 70   // 너무 빠름
                    : 100;               // 정상, 캡 없음
  const adjustedFluency = Math.min(azure.scores.fluency, fluencyCap);

  // 4. 가중 평균 (발음35 · 유창성35 · 억양20 · 의미오류10)
  const errorScore = (1 - Math.min(1, errorRate * 5)) * 100;
  const raw = accuracy             * 0.35
           + adjustedFluency      * 0.35
           + azure.scores.prosody * 0.20
           + errorScore            * 0.10;

  // 5. 100점 → 4점 척도 (0.5 단위)
  return toHalfPoint(raw);
}

핵심 원칙: 문법·발음 오류는 내용 영역에서 평가하지 않는다. 언어 영역(Ⅰ)에서 Azure 원시 데이터로 별도 채점. GPT가 받는 Transcript는 STT 언어모델이 보정한 텍스트이므로 여기서 문법 오류를 세는 것은 신뢰도가 보장되지 않는다. GPT 프롬프트에 명시적 금지 조항 포함.
신뢰도 확보 — 3회 실행 중앙값
async function scoreContentReliable(transcript, topic) {
  // 동일 입력 3회 실행 (temperature=0)
  const runs = await Promise.all([
    callGPT(transcript, topic),
    callGPT(transcript, topic),
    callGPT(transcript, topic),
  ]);

  const scores = runs.map(r => r.score).sort((a,b) => a-b);
  const median = scores[1]; // 3개 중 중앙값

  // 편차 기반 신뢰도 판정
  const spread = scores[2] - scores[0];
  const confidence = spread === 0   ? 'high'
                    : spread <= 0.5 ? 'medium'
                    :               'low';  // spread > 1.0 → 인간 검토

  return { ...runs[1], score: median, spread, confidence };
}

위협 1

위협 2
제스처 "적정 수준" 판단의 주관성
심사기준: "너무 많으면 산만, 없으면 경직"
대응: None(없음) / Appropriate(적정) / Excessive(과다) 3단계만. "적정"과 "없음" 사이 주관 판단 제거.
위협 3
수줍은 학생의 표정 부족으로 과도한 감점
심사기준: "수줍어도 끝까지 안정적이면 좋은 평가"
대응: 발표 완수(V7)를 독립 항목으로 분리. 표정 항목 비중 축소, 완수 시 보너스 구조.
위협 4 (핵심)
"능숙한 전달" — 텍스트만으로 판별 불가
대응: 4개 프록시 신호 교차 판단. Gemini 시선 패턴 + Azure 끊김 패턴 + WPM 편차 + 실수 후 회복률 합산.

Gemini 평가 항목 (V1–V7)

  • V1 시선 처리 — 카메라/청중을 의식하며 시선을 유지하는가? (Yes / Partial / No)
  • V2 표정·자세 — 내용에 어울리는 자연스러운 표정·안정적 자세인가? (Yes / Partial / No)
  • V3 제스처 — (None / Appropriate / Excessive)
  • V4 목소리 성량·또렷함 — 충분히 들리고 또렷한가? (Yes / Partial / No)
  • V5 능숙한 전달 — 내용을 소화해 말하는가? 대본 읽는 느낌인가? (Yes / Partial / No)
  • V6 완급(속도·멈춤·강조) — 핵심에서 속도 조절·강조가 있는가? (Yes / Partial / No)

Section 07

인간 검토 트리거 기준

needsHumanReview = true 가 되는 조건. 아래 중 하나라도 해당하면 심사위원 직접 검토.

1
confidence.overall === 'low'
3개 엔진 중 하나라도 신뢰도가 낮을 때. GPT spread > 1.0, Azure 음질 불량, Gemini 얼굴 미노출 등.
2
주요 플래그 존재
얼굴미노출_50%이상 / 음질불량 / 발표미완수 / 주제이탈_가능성 / 다중화자감지 / 대본낭독강_의심
3
영역간 점수 극단 괴리 — |lang - delivery| > 2.0점
예: 언어 4.0 + 전달 1.5 (유창하게 읽지만 완전히 경직된 경우) — 이상값 가능성 확인 필요.
4
GPT 내용 점수 spread > 1.0
3회 실행에서 점수 범위가 1점 이상이면 내용 판단이 불안정한 것. 해당 발표 직접 읽기 필요.
예상 인간 검토 비율: confidence HIGH (약 65%, 1,040명) · MEDIUM (약 25%, 400명) · LOW (약 10%, 160명). HIGH는 클릭 확인 1분, MEDIUM은 영상 일부 시청 3분, LOW는 전체 시청 5~7분. 심사위원 3명 × 2일이면 처리 가능한 규모.

핵심 요약

Azure Speech
Ⅰ. 언어
  • AccuracyScore · FluencyScore · ProsodyScore
  • 문법 오류는 wordDetails 원시 데이터로 측정
  • WPM 이상 시 Fluency 상한 캡
  • 가중 평균 → 4점 척도 (0.5 단위)
GPT-4o
Ⅱ. 내용
  • 발표 유형 4종 자동 분류
  • 유형별 체크리스트 분기 적용
  • temperature=0 · 3회 실행 중앙값
  • 보정 Transcript 사용, 문법 채점 금지
Gemini 2.0 Flash
Ⅲ. 전달
  • V1–V7 항목 체크리스트
  • 능숙한 전달: 4개 신호 교차 판단
  • 영상 품질 사전 체크 필수
  • 제스처 3단계만 (주관성 제거)

AI가 하는 것

  • Ⅰ 언어 4점 척도 산출
  • Ⅱ 내용 4점 척도 산출
  • Ⅲ 전달 4점 척도 산출
  • 점수 근거 텍스트 생성 (한국어)
  • 학부모·교사용 피드백 생성
  • 신뢰도 플래그 → 검토 트리거

AI가 하지 않는 것

  • 가중치 적용
  • 순위 산정
  • 수상 판단
  • 영역간 중요도 비교
  • 최종 합산 결정
가중치·수상 판단은 심사위원이
내부 논의를 통해 결정한다.
